# OLX Price Tracker

Сервис для автоматического отслеживания изменения цен на объявления OLX с уведомлением пользователей по электронной почте.

## Архитектура системы

### 1. Подписка на обновление
Процесс инициации отслеживания пользователем:
`[ Пользователь ]` --(POST `/api/subscribe` + URL, Email)--> `[ Controller ]`

**Логика обработки:**
1. Проверка: существует ли уже такой URL в БД (`table: ads`).
   - **Нет:** Создаем запись в `ads` (сохраняем url и текущую цену).
2. Создаем запись в `ad_user` (связь между объявлением и email).
3. `[ Mailer ]` отправляет письмо с подтверждением подписки.

### 2. Отслеживание (Cron / Scheduler)
Автоматизированный процесс проверки цен:
`[ Laravel Scheduler ]` --(Ежечасно)--> `[ TrackPricesCommand ]`

**Логика выполнения:**
1. Получение всех уникальных URL из таблицы `ads`, которые требуют проверки.
2. `[ OLX Fetcher Service ]` --(HTTP GET)--> `[ OLX Web Site ]`.
3. **Если цена изменилась:**
   - Обновляется поле `current_price` в таблице `ads`.
   - Выбираются все подписчики для этого `ad_id`.
   - `[ Queue ]` --> `[ Mailer ]` --> Асинхронная отправка уведомлений пользователям.

## 🛠 Выбор стратегии получения данных (Trade-offs)

Для реализации парсинга данных с OLX были рассмотрены три варианта:

### Вариант А: Парсинг HTML-страницы (Web Scraping)
- **Метод:** GuzzleHTTP + Symfony DOMCrawler (CSS селекторы).
- **Плюсы:** Простой старт.
- **Минусы:** Крайне хрупкий подход. OLX использует динамические CSS-классы, структура DOM нестабильна. Высокий риск блокировок.
- **Вердикт:** ❌ Не рекомендуется.

### Вариант Б: Использование API (Official / Mobile)
- **Метод:** Реверс-инжиниринг запросов мобильного приложения.
- **Плюсы:** Высокая скорость, структурированные данные.
- **Минусы:** Официальный API закрыт для частных лиц. Мобильный API требует генерации динамических токенов подписи, что избыточно для данной задачи.
- **Вердикт:** ❌ Слишком сложно для реализации.

### Вариант В: Извлечение SSR-состояния (ОБРАННЫЙ ВАРІАНТ)
- **Метод:** Извлечение JSON-объекта состояния страницы (Server-Side Rendering State) из HTML-кода (переменная `window.__PRERENDERED_STATE__`) с помощью регулярных выражений.
- **Преимущества:**
    - **Надежность:** Структура JSON данных меняется значительно реже, чем визуальная верстка сайта.
    - **Простота:** Не требуются API-ключи, работа ведется через обычные GET-запросы.
    - **Точность:** Получение "чистых" числовых данных без необходимости очистки текста от валютных символов и пробелов.
- **Вердикт:** ✅ **Оптимальный выбор.**

## 🏗 Технический стек
- **Backend:** PHP 8.2+, Laravel 11.
- **Database:** PostgreSQL/MySQL.
- **Queue:** Redis (для обработки очередей писем).
- **Environment:** Docker (Laravel Sail).
- **Testing:** PHPUnit (Feature & Unit tests).

Для обеспечения гибкости реализован `PriceFetcherInterface`. Это позволяет заменить логику парсинга (например, на официальный API в будущем) без изменения основного кода команды трекера.
